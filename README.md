# PrivacyEscrowZK

Merchant privacy for the Machine Payments Protocol (MPP). A drop-in escrow replacement that uses Poseidon-based note commitments and Groth16 ZK-SNARKs so merchants can receive payments without revealing their identity on-chain.

Deployed on **Tempo mainnet** (Chain 4217).

| Contract | Address |
|---|---|
| PrivacyEscrowZK | `0x472d53B0F5d3b8e5E6f03859961a0db63a97fA9d` |
| Groth16Verifier | `0x48f22b5C31695B499B5a3520A59742F66596abd9` |

## How it works

Standard MPP escrows pay merchants directly — the entire payer→merchant graph is public. PrivacyEscrowZK replaces direct payouts with an intermediate note layer:

```
1. OPEN        Payer deposits tokens. Payee sets merchantCommitment = Poseidon(pubKey, blinding).
2. SETTLE      Each voucher settlement mints a Poseidon note commitment into an incremental Merkle tree.
               Funds stay in the shared escrow pool.
3. REDEEM (ZK) Merchant generates a Groth16 proof of note ownership and redeems to a fresh address.
               Only the proof, nullifier, amount, asset, and recipient hit the chain.
```

The payer→merchant link is cryptographically broken. Clients sign the same EIP-712 vouchers as standard MPP — they are unaware the privacy layer exists.

### What's hidden

| Data | Standard Escrow | PrivacyEscrowZK |
|---|---|---|
| Payer identity | Visible | Visible |
| Deposit amounts | Visible | Visible |
| Which merchant | Visible | **Hidden** |
| Merchant payout address | Visible | **Hidden** |
| Payer → merchant link | Visible | **Hidden** |

Privacy strength scales with the number of merchants sharing the same escrow instance.

## ZK circuit

`circuits/note_redeem.circom` — ~5,700 constraints, sub-second proof generation.

**Public inputs:** nullifier, merkleRoot, amount, asset, recipient

**Private inputs:** merchantPubKey, blinding, noteRandomness, channelId, pathIndices[20], siblings[20]

The circuit proves:

1. `merchantCommitment = Poseidon(merchantPubKey, blinding)`
2. `noteCommitment = Poseidon(asset, amount, merchantCommitment, channelId, noteRandomness)`
3. `nullifier == Poseidon(noteCommitment, merchantPubKey)` — prevents double-spend
4. `merkleRoot == MerkleProof(noteCommitment, siblings, pathIndices)` — note exists in tree
5. `recipient` is constrained — prevents front-running proof substitution

## Per-channel smart accounts

The escrow requires `msg.sender == payee` for `setMerchantCommitment()` and `close()`. If the merchant used their real address, the privacy would be broken at channel-open time.

`ChannelPayeeFactory` solves this by deploying a disposable `ChannelPayee` contract per session via CREATE2:

1. Merchant generates a **one-time keypair** for the session
2. `Factory.computePayeeAddress(key)` returns a deterministic address (before deploy)
3. Server tells payer to open the channel to that address
4. `Factory.deploy(key)` materializes the contract (anyone can call)
5. Anyone can relay merchant-signed instructions to the contract (setCommitment, close)

The on-chain observer sees `payee = 0x8a3f…` — a disposable contract with no link to the merchant's real identity.

## Build & test

Requires [Foundry](https://getfoundry.sh/) (Tempo fork: `foundryup -n tempo`).

```bash
# Build contracts
forge build

# Run tests
forge test -vvv
```

### Circuits

Requires [circom](https://docs.circom.io/getting-started/installation/) and Node.js.

```bash
cd circuits
npm install

# Compile circuit
circom note_redeem.circom --r1cs --wasm --sym -o build

# Trusted setup (already done — build/ contains final zkey)
snarkjs groth16 setup build/note_redeem.r1cs build/pot14_final.ptau build/note_redeem_0000.zkey
snarkjs zkey contribute build/note_redeem_0000.zkey build/note_redeem_final.zkey

# Generate Solidity verifier
snarkjs zkey export solidityverifier build/note_redeem_final.zkey ../src/Groth16Verifier.sol

# Generate test proof
node gen_test_proof.mjs
```

## Mainnet deployment

```bash
# Deploy contracts (requires Tempo wallet access key)
bash script/mainnet-deploy.sh
```

The deploy script uses `forge create` with Tempo's passkey-based smart account (`--tempo.access-key`, `--tempo.root-account`, `--tempo.fee-token`). It deploys PrivacyEscrow and ChannelPayeeFactory, saving addresses to `script/mainnet-addrs.env`.

For PrivacyEscrowZK, the Groth16Verifier and Poseidon libraries must be deployed first, then the escrow is constructed with the verifier address.
